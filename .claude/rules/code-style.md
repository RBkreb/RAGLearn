# Python 代码组织与风格底线规则

本规则规定了 Python 代码在结构、风格和错误处理上的强制最低标准，以保持可读性、可维护性与一致性。

## 1. 文件大小

- 单个 Python 文件 **不应超过 500 行**（不含自动生成的代码、空行和注释）。
- 超过 500 行的文件 **必须** 根据功能拆分为多个模块/包。

## 2. 函数与方法长度

- 单个函数或方法 **不应超过 50 行**（不含空行和注释）。
- 超过 50 行表明存在职责过多风险，**必须** 拆分为多个更小的函数。

## 3. 类长度

- 单个类 **不应超过 300 行**。
- 超过 300 行通常意味着承担过多责任，应当拆分为多个协作类。

## 4. 代码复杂度

- 单个函数的 **圈复杂度（McCabe）** 不得超过 **10**。
- 可使用 `radon` 或 `pytest --mccabe` 检测，超过时 **必须** 重构。

## 5. 错误处理

- **使用具体异常类型**：禁止使用裸 `except:`，应捕获 `ValueError`、`TypeError`、`KeyError` 或自定义异常。
- **raise 语句必须提供有意义的错误信息**：例如 `raise ValueError("amount must be positive")`。
- **不允许吞掉异常**：捕获后必须记录日志，或重新抛出自定义异常，不得忽略（`pass` 仅在非常有限的场景如资源清理时使用）。
- **资源管理**：对于文件、网络连接、锁等资源，必须使用 `with` 语句或 `try/finally` 确保释放。
- 函数返回值应类型一致；如需表达错误，优先抛出异常而非返回特殊值（如 `None`）。

## 6. 类型注解

- **所有公共函数、方法** 必须包含参数类型注解和返回值类型注解。
- 私有函数（以下划线开头）建议添加注解。
- 复杂项目 **应** 通过 `mypy --strict` 检查。
- 简单变量（循环索引等）可不注解，但类属性应标注类型。

## 7. 命名规范

遵循 [PEP 8](https://peps.python.org/pep-0008/) 命名约定：

- **变量、函数、方法**：`snake_case`
- **常量**：`UPPER_SNAKE_CASE`
- **类**：`PascalCase`
- **私有成员**：单下划线开头，如 `_internal_method`
- **避免单字母名称**（除极简循环中的 `i`, `j`, `k` 和 lambda 参数）
- **布尔变量/函数**：使用 `is_`, `has_`, `should_` 前缀（如 `is_valid`, `has_permission`）

## 8. 导入规范

- 导入按以下顺序分组，每组之间空一行：
  1. 标准库模块
  2. 第三方库模块
  3. 本地应用/模块
- 每组内按 **字母顺序** 排序。
- **禁止** 使用 `from module import *`（通配符导入）。
- 避免相对导入（除非在包内部短路径）；优先使用绝对导入。

## 9. 文档字符串（Docstring）

- 公共模块、类、函数、方法 **必须** 有 docstring。
- 推荐使用 **Google 风格** 或 **NumPy 风格**，至少包含：
  - 简要描述功能
  - 参数（`Args:`）
  - 返回值（`Returns:`）
  - 可能抛出的异常（`Raises:`）
- 示例（Google 风格）：
  ```python
  def fetch_data(user_id: int, timeout: float = 30.0) -> dict:
      """根据用户 ID 获取数据。

      Args:
          user_id: 用户的唯一标识符。
          timeout: 请求超时时间（秒）。

      Returns:
          包含用户信息的字典，键为 'id', 'name', 'email'。

      Raises:
          ValueError: 当 user_id 小于 0 时。
          ConnectionError: 当无法连接到 API 时。
      """
