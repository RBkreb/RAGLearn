# Python 测试底线规则

本规则定义了 Python 项目中测试的强制性要求，确保代码质量、可维护性与关键路径的可靠性。

## 1. 测试驱动开发 (TDD) 流程

- **实现**：编写最少代码使测试通过。
- **提交**：测试通过后，方可提交代码。

## 2. 覆盖率门槛

- **整体行覆盖率**：不低于 **80%**。
- **关键模块**（核心业务逻辑、安全相关、数据校验、加密/解密、支付处理等）：不低于 **90%**。
- 持续集成（CI）中若覆盖率未达标，**必须** 构建失败。

## 3. 测试类型与职责分离

- **单元测试**（`test/unit/`）：针对单个函数/类，隔离外部依赖（使用 mock）。
- **集成测试**（`test/integration/`）：验证模块间交互、数据库、外部 API 等真实集成。
- **端到端（E2E）测试**（`test/e2e/`）：覆盖完整用户场景或关键业务路径。

## 4. 关键路径必须包含 E2E 测试

**关键路径** 包括但不限于：

- 主要工作流
- 跨多个服务或数据库的流程
- 核心 API
- 数据处理管道的重要阶段

**要求**：每个关键路径至少有一个 E2E 测试用例。E2E 测试应尽可能接近生产环境，使用独立的测试数据库/容器。

## 5. 测试代码规范

- 测试文件命名：`test_*.py`
- 测试函数命名：`test_<描述>`，描述清楚场景与预期行为（例如 `test_deposit_positive_amount_increases_balance`）。
- 使用 **pytest** 框架，充分利用 `fixture` 和 `parametrize`。
- 测试不可依赖执行顺序，必须隔离且幂等。
- 每个测试方法只验证一个行为。

## 6. Mock 与集成使用原则

- 单元测试中对外部系统（网络、数据库、文件系统）**必须** 使用 mock（`unittest.mock` 或 `pytest-mock`）。
- 集成测试 **应尽量** 使用真实依赖的测试实例（如测试数据库容器、本地 stub 服务），避免过度 mock 导致失真。
- 对于 E2E 测试，仅 mock 不可控的外部第三方服务。

## 7. 测试执行与 CI

- CI 管道 **必须** 运行所有单元测试、集成测试和关键 E2E 测试。
- 测试失败时 **必须** 中断流水线，不允许合并。
- 提供快速反馈：单元测试应在数分钟内完成。

## 8. 文档化测试

- 每个模块的测试策略应在 `README` 或 `tests/README.md` 中简要说明。
- 复杂的测试 fixture 或 setup 需要有注释说明目的。

## 9. 禁止事项

-  禁止将生产环境的真实数据拷贝到测试中（需匿名化或生成模拟数据）。
-  禁止在测试中依赖外部网络服务而不进行适当 mock 或 stub。
-  禁止为了覆盖率而编写无断言或无效断言的测试用例。
